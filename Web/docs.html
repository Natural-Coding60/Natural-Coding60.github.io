Introduction

Welcome to the official Nexus documentation. Nexus is an open-source build system and developer platform designed to help teams build, test, and deploy modern software faster.

Whether you're a solo developer working on a single-repo project or part of a large team managing a complex monorepo, Nexus provides the tools to streamline your workflow and eliminate repetitive build work.

💡

New to Nexus? Start with the Quick Start guide to get a project running in under 5 minutes.

Installation

The recommended way to install Nexus is via npm. It's available as a single global CLI binary with no platform-specific dependencies.

Install Nexus CLI
# npm (recommended) npm install -g nexus-app # pnpm pnpm add -g nexus-app # homebrew (macOS / Linux) brew install nexus-app

Verify the installation by checking the version:

Verify
nexus --version # nexus/3.0.4 darwin-arm64 node-v20.11.0

Quick Start

Creating a new project is a single command. Nexus will prompt you to choose a template, set up your project, and install dependencies automatically.

Create a new project
# Interactive nexus init # With template nexus init my-app --template react-ts # From existing repo cd my-existing-project nexus init --detect

Once initialized, start the development server:

Start dev server
cd my-app nexus dev # Output: # ⚡ nexus v3.0.4 # ✓ Ready in 380ms # ➜ http://localhost:3000

Configuration

Nexus is zero-config by default, but you can customize it via a nexus.config.ts file in your project root.

nexus.config.ts
import { defineConfig } from 'nexus-app'; export default defineConfig({ build: { outDir: 'dist', target: 'esnext', minify: true, sourcemap: true, }, dev: { port: 3000, hmr: true, open: false, }, cache: { remote: true, // Requires Pro local: '~/.nexus/cache', }, plugins: [ reactPlugin(), typeScriptPlugin(), ], });

Tasks

Tasks are the atomic units of work in Nexus. Each task is a shell command with optional dependencies, inputs, and outputs declared in your nexus.config.ts.

Declaring inputs and outputs allows Nexus to cache task results and skip re-execution when nothing has changed.

Defining tasks
export default defineConfig({ tasks: { build: { command: 'tsc && vite build', inputs: ['src/**', 'tsconfig.json'], outputs: ['dist/**'], cache: true, }, test: { command: 'vitest run', dependsOn: ['build'], inputs: ['src/**', 'tests/**'], }, deploy: { command: './scripts/deploy.sh', dependsOn: ['build', 'test'], }, }, });

Task Graph

Nexus builds a directed acyclic graph (DAG) of your tasks based on their dependsOn declarations. It executes tasks in topological order and runs independent tasks in parallel whenever possible.

You can visualize your task graph at any time:

Visualize
# Open task graph in browser nexus graph # Output to SVG nexus graph --output graph.svg

Caching

Nexus uses a content-addressable cache based on a hash of a task's inputs, environment variables, and the Nexus version. If the hash matches a cached result, the task is replayed from cache instantly.

✓

Cache hit rate in the wild: Teams with well-declared task inputs/outputs typically see 70–95% cache hit rates in CI, reducing pipeline time by 4–10×.

Local cache

Local cache is stored at ~/.nexus/cache by default. You can change this location in your config. The local cache is shared across all projects on the machine.

Remote cache

Remote cache (Nexus Pro) allows your entire team and CI runners to share a single cache. Artifacts are stored in our cloud (or your own S3-compatible bucket) and fetched on demand.

Monorepos

Nexus has first-class support for monorepos using npm workspaces, yarn workspaces, or pnpm. It automatically discovers packages and builds a cross-package task graph.

Monorepo workspace config
// nexus.config.ts (root) export default defineConfig({ workspaces: { packages: ['packages/*', 'apps/*'], tasks: { build: { dependsOn: ['^build'] }, // ^ means deps first test: { dependsOn: ['^build'] }, }, }, });

To build only the packages affected by your latest changes:

Affected packages
# Build only what changed since main nexus build --affected --base=main # Print affected packages (dry run) nexus affected --base=main

nexus init

Initializes a new Nexus project or adds Nexus to an existing project.

nexus init flags
FlagDefaultDescription
--template <name>interactiveStarter template to use
--detectfalseAuto-detect existing framework
--no-installfalseSkip dependency installation

nexus build

Runs the build task (or a custom task) for the project. Uses the local cache by default.

nexus build flags
FlagDefaultDescription
--affectedfalseOnly build affected packages
--no-cachefalseSkip the cache entirely
--remotefalseUse remote cache (Pro)
--concurrency <n>CPU countMax parallel tasks
--verbosefalsePrint full task output

nexus dev

Starts a local development server with hot module replacement enabled.

nexus test

Runs your test suite. Supports Jest, Vitest, Mocha, and any custom test runner declared in your config.

nexus deploy

Builds and deploys your application. Target is inferred from your config or can be specified explicitly.

Deploy examples
# Deploy to configured target nexus deploy # Deploy to Vercel nexus deploy --target vercel # Deploy to custom server nexus deploy --target ssh --host user@my-server.com

nexus.config.ts — Full Reference

All available top-level configuration keys are listed below. Every key is optional; Nexus will infer sensible defaults for everything not declared.

Full config skeleton
import { defineConfig } from 'nexus-app'; export default defineConfig({ root: '.', srcDir: 'src', outDir: 'dist', build: { target: 'esnext', minify: true, sourcemap: 'inline', splitting: true, treeshake: true, }, dev: { port: 3000, https: false, hmr: true, proxy: { '/api': 'http://localhost:8080', }, }, test: { runner: 'vitest', coverage: true, threshold: 80, }, cache: { local: '~/.nexus/cache', remote: false, ttl: '7d', }, plugins: [], });

Environment Variables

Nexus reads .env, .env.local, and .env.<mode> files. Variables prefixed with NEXUS_PUBLIC_ are exposed to the client-side bundle.

Built-in Nexus environment variables
VariableDescription
NEXUS_MODEdevelopment, production, or test
NEXUS_TOKENAuthentication token for remote cache (Pro)
NEXUS_CACHE_URLCustom remote cache endpoint
NEXUS_CONCURRENCYOverride max parallel tasks
NEXUS_LOG_LEVELsilent, info, warn, debug
NEXUS_NO_COLORDisable color output (CI friendly)

Plugins

Nexus's plugin system allows you to hook into the build lifecycle, transform files, add new CLI commands, and more. The community plugin registry contains over 800 plugins.

Using a plugin
import { reactPlugin } from '@nexus/plugin-react'; import { sentryPlugin } from 'nexus-plugin-sentry'; export default defineConfig({ plugins: [ reactPlugin({ jsxRuntime: 'automatic' }), sentryPlugin({ dsn: process.env.SENTRY_DSN }), ], });

CI/CD Integration

Nexus works with any CI provider. Here are examples for the most common ones.

.github/workflows/ci.yml
name: CI on: [push, pull_request] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: { node-version: '20' } - uses: nexuscorp/setup-nexus@v3 - run: nexus build --affected --remote env: NEXUS_TOKEN: ${{ secrets.NEXUS_TOKEN }}
.gitlab-ci.yml
image: node:20 stages: [build, test] build: stage: build script: - npm install -g nexus-app - nexus build --remote variables: NEXUS_TOKEN: $NEXUS_TOKEN
.circleci/config.yml
version: '2.1' jobs: build: docker: [{ image: cimg/node:20.0 }] steps: - checkout - run: npm install -g nexus-app - run: nexus build --remote

Plugin API

Plugins are functions that receive a Nexus context object and can register hooks, add resolvers, or extend the CLI.

Writing a plugin
import type { NexusPlugin } from 'nexus-app'; export function myPlugin(options = {}): NexusPlugin { return { name: 'my-plugin', setup(build) { // Transform .txt files build.onLoad({ filter: /\.txt$/ }, async (args) => { const text = await fs.readFile(args.path, 'utf8'); return { contents: `export default ` + JSON.stringify(text), loader: 'js', }; }); }, }; }

Lifecycle Hooks

Nexus exposes hooks at each stage of the build pipeline. All hooks support async functions.

Nexus lifecycle hooks
HookWhen it runs
onStartBefore any task begins
onLoadWhen a module is resolved
onTransformAfter a module is loaded, before bundling
onBundleEndAfter the bundle is written to disk
onTaskCompleteAfter each individual task completes
onEndAfter all tasks complete (success or failure)
onErrorWhen any task throws an error

Migrating from Webpack

Nexus ships an automatic migration tool that reads your existing Webpack, Vite, or CRA configuration and produces an equivalent Nexus config with no manual intervention required for most projects.

Migrate
# Auto-detect and migrate nexus migrate # From a specific tool nexus migrate --from webpack nexus migrate --from vite nexus migrate --from cra
⚠️

Always commit your existing config to version control before running nexus migrate. Review the generated nexus.config.ts before deleting your old config files.
